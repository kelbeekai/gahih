# 가힣 테스트 우선순위와 UI 개선 분리

## 1. 우선순위 기준

### P0: 배포 전 반드시 통과

서비스 기본 사용과 보안/데이터 무결성에 직접 영향을 주는 항목이다.

- 애플리케이션 실행
- 루트/국가 커뮤니티 접근
- 회원가입/로그인/로그아웃
- 게시글 작성/상세/수정/삭제
- 댓글 작성/수정/삭제
- 첨부파일 업로드/다운로드/미리보기
- 관리자 접근 제한
- 관리자 핵심 조치
- 푸터 정책 페이지 접근
- Git clean/build 성공

### P1: 배포 전 강하게 권장

운영 품질과 실제 서비스 정책에 영향을 주지만, 일부 보류 가능성이 있는 항목이다.

- 신고 전 과정
- 비밀글 정책
- 회원 정지/탈퇴 정책
- 국가별 데이터 분리
- 마이페이지 세부 기능
- 관리자 로그/신고자 활동
- 닉네임 이력/예약 확인

### P2: 포트폴리오 완성도 개선

치명 오류는 아니지만 화면 완성도와 설명력에 영향을 주는 항목이다.

- 목록 UI 정렬
- 관리자 테이블 가독성
- 모바일 화면
- 문구 자연스러움
- 오류 페이지 문구
- README/문서 최신화

## 2. 추천 테스트 순서

### 1차: 30~60분 핵심 생존 테스트

1. prod-like 실행.
2. `/`, `/c/DE`, `/c/JP` 접근.
3. 회원가입 모달/정책 페이지 확인.
4. 로그인/로그아웃.
5. 게시글 작성/상세/수정/삭제.
6. 댓글 작성/수정/삭제.
7. 첨부파일 1개 업로드/미리보기/다운로드.
8. 관리자 로그인/게시글 블라인드/복구.
9. Git clean 확인.

### 2차: 반나절 핵심 회귀 테스트

1. 회원가입 이메일 인증.
2. 아이디 찾기/비밀번호 재설정.
3. 마이페이지 프로필/비밀번호 변경.
4. 게시글 검색/정렬/페이지네이션.
5. 첨부파일 제한 3개/10MB/30MB.
6. 댓글 정렬/페이지네이션.
7. 좋아요/싫어요 취소/전환.
8. 신고 접수/중복 신고 차단.
9. 관리자 게시글/댓글/회원/신고/로그.
10. 정지/탈퇴 회원 접근 제한.

### 3차: 포트폴리오 제출 전 마감 테스트

1. 독일/일본 데이터 분리.
2. 비밀글 접근 권한.
3. 관리자 로그 스냅샷 모달.
4. 푸터 정책 원문 전체 확인.
5. 오류 페이지.
6. 모바일 폭에서 주요 화면 확인.
7. README/문서 최신화.
8. 최종 빌드/실행/커밋 clean.

## 3. 화면 보면서 고칠 수 있는 UI/문구/정렬 포인트

아래 항목은 테스트 중 발견하면 바로 고쳐도 되지만, P0 기능 테스트를 방해할 정도로 깊게 들어가지는 않는다.

### 3.1 회원가입/인증

- 필수 동의 모달 본문이 너무 길어 보이는지.
- 모달 내부 체크 문구가 자연스러운지.
- 인증번호 발송/검증 버튼 위치가 직관적인지.
- 오류 문구가 한 화면에서 잘 보이는지.

### 3.2 게시글 목록

- 검색/정렬 카드가 너무 크거나 눈에 띄는지.
- 정렬/목록 수 드롭다운과 검색창 간 간격이 어색한지.
- 긴 제목, 긴 닉네임, 댓글 수, 첨부 아이콘의 수직 정렬이 자연스러운지.
- 고정글과 일반글의 구분이 과하지도 약하지도 않은지.

### 3.3 게시글 상세

- 이미지 미리보기 크기가 적절한지.
- 첨부파일 메타 정보가 너무 산만하지 않은지.
- 좋아요/싫어요 버튼이 본문과 댓글 사이에서 잘 보이는지.
- 이전/다음 글과 하단 목록이 과하게 길어 보이지 않는지.

### 3.4 댓글

- 댓글 작성 폼과 댓글 목록 사이 간격이 적절한지.
- 댓글 수정/삭제/신고 버튼이 너무 붙어 있지 않은지.
- 댓글 정렬/페이지네이션 위치가 직관적인지.
- 멘션 표시가 본문을 방해하지 않는지.

### 3.5 신고 UI

- 신고 드롭다운이 화면 밖으로 나가지 않는지.
- 신고 사유 선택과 제출 버튼이 명확한지.
- 이미 신고한 대상의 UI가 어색하지 않은지.

### 3.6 마이페이지

- 통계 카드 높이/간격이 일정한지.
- 최근 반응 카드가 비어 있을 때도 화면이 어색하지 않은지.
- 내가 쓴 글 버튼이 충분히 눈에 띄는지.
- 정지/탈퇴 상태 안내 문구가 딱딱하지 않은지.

### 3.7 관리자 화면

- 관리 드롭다운이 테이블 행과 겹치지 않는지.
- 사유 입력 prompt/confirm 흐름이 헷갈리지 않는지.
- 관리자 로그 스냅샷 모달이 너무 작거나 넓지 않은지.
- 회원 목록에서 위험 조치 버튼이 너무 쉽게 눌릴 것처럼 보이지 않는지.

### 3.8 정책/문서 페이지

- 개인정보 처리방침 표가 읽기 좋은지.
- 이용약관/운영정책 본문 줄 간격이 충분한지.
- 면책조항/운영문의가 너무 비어 보이지 않는지.
- 푸터 링크 문구가 실제 페이지 제목과 일치하는지.

## 4. 보류해도 되는 개선

아래 항목은 배포 전 필수는 아니다.

- 완전한 모바일 최적화.
- 다크모드.
- 일출/일몰 자동 테마.
- 날씨/환율/추천글 티저.
- 국가별 로고 애니메이션.
- 홈 국기 펄럭임 애니메이션.
- 프론트 프레임워크 전환.
- 전체 자동 테스트 커버리지 확보.

## 5. 배포 전 최소 수정 권장 사항

현재 소스 기준으로 기능 자체보다 문서/실행 경험에서 다음을 확인하는 것이 좋다.

1. 루트 `README.md`가 초기 계획 상태에 가까우므로, 포트폴리오 제출 전 현재 구현 상태로 갱신한다.
2. `docs/00-overview/project-overview.md`의 구현 체크박스가 오래된 상태라면 갱신한다.
3. `application-dev.properties`가 H2 TCP 모드이므로, README에 H2 서버 필요 여부를 명확히 쓰거나 dev를 embedded/file H2로 바꾸는지 결정한다.
4. 자동 테스트는 전체를 새로 만들기보다 첨부파일 제한, 닉네임 정책, 신고 중복, 권한 정책처럼 핵심 정책 로직부터 최소 추가하는 방향이 현실적이다.
