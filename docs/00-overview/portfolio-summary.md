# 가힣 포트폴리오 소개

## 1. 프로젝트 한 줄 소개

가힣는 해외 거주 한인과 한국에 관심 있는 사용자를 위한 국가별 커뮤니티 서비스입니다.

단순 게시판 CRUD를 넘어, 회원 상태, 이메일 인증, 신고, 관리자 조치, 운영 정책, 개발/운영 환경 분리, HTTPS 배포, 운영 로그 점검까지 직접 다룬 Spring Boot 기반 포트폴리오 프로젝트입니다.

- 운영 주소: https://gahih.com
- 현재 운영 배포 버전: `v0.1.8`
- 초기 국가 커뮤니티: 독일(DE), 일본(JP)
- 배포 환경: AWS Lightsail + Ubuntu + Nginx + MySQL + systemd + Let’s Encrypt

## 2. 프로젝트 목적

이 프로젝트는 “작지만 실제 운영 가능한 커뮤니티 서비스”를 목표로 개발했습니다.

단순히 게시글을 작성하고 조회하는 기능보다, 실제 서비스를 운영할 때 발생하는 문제를 직접 다루는 데 초점을 두었습니다.

주요 설계 고민은 다음과 같습니다.

- 회원이 탈퇴하거나 정지되었을 때 기존 콘텐츠를 어떻게 처리할 것인가
- 비밀글은 누구에게 어디까지 보여줄 것인가
- 신고 대상과 신고자를 어떻게 기록하고 관리할 것인가
- 관리자 조치는 어떤 단위로 기록하고 추적할 것인가
- 게시판별로 댓글, 반응, mention 정책을 어떻게 다르게 적용할 것인가
- 개발 환경과 운영 환경을 어떻게 분리하고 검증할 것인가
- 실제 배포 환경에서 비밀값, DB, 파일 업로드, HTTPS, 로그를 어떻게 관리할 것인가
- 공개 인터넷 노출 후 발생하는 봇/스캐너 요청과 방문자 통계 오염을 어떻게 처리할 것인가

## 3. AI 활용 방식

이 프로젝트는 ChatGPT의 도움을 적극적으로 받아 설계와 구현을 진행한 AI-assisted 포트폴리오 프로젝트입니다.

프로젝트를 기획하고, 어떤 서비스를 만들 것인지 고민하며, 구현된 기능을 직접 실행하고 검증하면서 코드 구현, 리팩토링 방향 제안, 운영 정책 설계 보조, 오류 원인 분석, 문서 작성 과정에는 ChatGPT의 도움을 활용했습니다.

이 프로젝트를 통해 처음부터 모든 코드를 혼자 작성하는 경험보다, 실제로 동작하는 서비스를 완성하고 배포 가능한 상태까지 끌고 가는 경험에 집중했습니다. 특히 요구사항을 정리하고, 구현 결과를 검증하며, 문제가 발생했을 때 원인을 추적하고 수정 방향을 판단하는 과정을 반복했습니다.

현재 이 프로젝트의 모든 구현 세부사항을 완전히 숙지했다고 말하기는 어렵습니다. 그래서 배포 이후에는 프로젝트 구조, 도메인 정책, Spring MVC 흐름, JPA 매핑, QueryDSL 검색, 인증/인가, 예외 처리, 배포 환경을 차례로 복습하며 제 것으로 만들어갈 계획입니다.

이 프로젝트는 AI를 개발 보조 도구로 활용해 실제 서비스를 끝까지 구현하고 검증해 본 첫 번째 실전 프로젝트입니다.

## 6. 중점 구현 포인트

### 6.1 회원 상태 기반 운영 정책

회원 상태를 `ACTIVE`, `SUSPENDED`, `WITHDRAWN`, `DELETED`로 구분하고, 상태에 따라 서비스 접근 범위를 다르게 처리했습니다.

정지 회원은 일반 활동을 제한하고, 탈퇴 회원은 유예 기간 이후 최종 종료될 수 있도록 설계했습니다. 탈퇴 또는 삭제된 회원의 콘텐츠와 닉네임 노출도 별도 정책으로 분리했습니다.

### 6.2 비밀글과 이용문의 정책

이용문의 게시판에서는 비밀글을 지원하고, 비밀글은 작성자와 관리자 중심으로 열람 권한을 제한했습니다.

비밀글을 볼 수 없는 사용자는 본문, 댓글, mention 후보를 볼 수 없도록 처리했습니다. 이를 통해 단순 화면 숨김이 아니라 서버에서 내려주는 데이터 자체를 제한하는 방향으로 구현했습니다.

### 6.3 게시판별 댓글 / 반응 / mention 정책

게시판 성격에 따라 댓글과 반응 정책을 다르게 적용했습니다.

예를 들어 구인구직 게시판은 사업주, 고용주, 구직자 간 분쟁과 비방을 줄이기 위해 댓글과 반응을 제한했습니다. 댓글이 허용되지 않는 게시판에서는 mention도 함께 비활성화되도록 정책을 통일했습니다.

mention 기능은 다음 정책을 반영했습니다.

- 댓글 허용 게시판에서만 mention 가능
- 구인구직 게시판에서는 mention 불가
- 비밀글에서는 열람 권한에 따라 mention 제한
- 일반 사용자의 관리자 mention은 이용문의 중심으로 제한
- 관리자는 댓글 허용 게시판에서 다른 관리자를 mention 가능
- 자기 자신 mention은 제한

### 6.4 신고와 관리자 조치

게시글, 댓글, 첨부파일, 작성자 신고를 분리해 처리했습니다.

신고는 대상별로 기록되며, 중복 신고를 제한했습니다. 관리자는 신고 내용을 확인한 뒤 블라인드, 복구, 삭제 등의 조치를 수행할 수 있고, 조치 사유와 당시 상태를 관리자 로그에 기록하도록 구현했습니다.

이를 통해 단순 관리자 기능이 아니라, 사후 추적이 가능한 운영 흐름을 구성했습니다.

### 6.5 개발 환경과 운영 환경 분리

개발 환경은 H2와 테스트 데이터를 기반으로 빠르게 화면과 기능을 확인할 수 있도록 구성했습니다.

운영 환경은 MySQL과 `prod` 프로필을 사용하며, Hibernate `ddl-auto=validate`로 운영 스키마와 엔티티 매핑이 일치하는지 검증하도록 구성했습니다.

비밀값은 환경변수로 분리했습니다.

- DB URL
- DB 계정
- 메일 계정
- 관리자 초기 계정
- 파일 업로드 경로

### 6.6 실제 운영 배포

단순 로컬 실행에서 끝내지 않고 실제 도메인과 HTTPS를 연결해 운영 가능한 형태로 배포했습니다.

배포 구성은 다음과 같습니다.

- AWS Lightsail Ubuntu 서버
- MySQL 운영 DB
- Spring Boot executable JAR
- systemd 서비스 등록
- Nginx reverse proxy
- Let’s Encrypt HTTPS 인증서
- `https://gahih.com` 대표 주소 고정
- HTTP 및 www 요청 redirect
- UFW 방화벽 설정
- DB 백업 및 Lightsail snapshot 생성

### 6.7 방문자 통계와 봇 요청 대응

공개 인터넷에 노출된 뒤 봇/스캐너 요청이 유입되면서 단순 공개 쿠키 기반 방문자 통계가 오염될 수 있었습니다.

이를 해결하기 위해 기본 방문자 집계 방식을 로그인 일반 회원 기준 일일 활동 지표로 변경했습니다.

- 로그인한 일반 회원만 한국 시간 기준 하루 1회 집계
- 관리자는 집계 제외
- property 변경으로 기존 공개 쿠키 기반 방식과 스위칭 가능
- 스캐너 경로는 Nginx에서 404로 차단

이 방식은 비회원 전체 유입량을 보여주지는 않지만, 초기 서비스 화면에 노출되는 숫자를 더 신뢰 가능한 회원 활동 지표로 유지하기 위한 선택입니다. 비회원 유입 분석은 향후 GA4 또는 Search Console 등 별도 분석 도구로 분리하는 방향을 고려하고 있습니다.

## 8. 문제 해결 사례

### 8.1 잘못된 path variable 요청 처리

`/c/DE/posts/abc`처럼 숫자 ID가 필요한 경로에 문자열이 들어오는 경우 기존에는 500 오류로 처리될 수 있었습니다.

이를 서버 장애가 아니라 잘못된 요청으로 분류하기 위해 `MethodArgumentTypeMismatchException`을 전역 예외 처리기에서 400 오류로 처리하도록 개선했습니다.

### 8.2 오류 페이지 헤더 컨텍스트 개선

오류 페이지에서는 로그인 상태와 국가 선택 드롭다운이 정상적으로 반영되지 않는 문제가 있었습니다.

전역 예외 처리기에서 오류 페이지용 공통 모델을 구성하도록 변경하고, 오류 템플릿에서 로그인 회원 정보를 헤더에 전달하도록 수정했습니다.

### 8.3 mention 정책 보강

관리자 mention, 비밀글 mention, 댓글 불가 게시판 mention 정책을 테스트 중 발견한 문제에 맞춰 보강했습니다.

화면에서 버튼을 숨기는 것뿐 아니라, 서버에서 mention 후보 생성과 mention 저장을 함께 제한해 우회 입력 가능성을 줄였습니다.

### 8.4 이메일 인증 타이머 오류

운영 환경에서 이메일 인증코드 발송 직후 만료 메시지가 잠깐 표시될 수 있었습니다.

원인은 서버가 `LocalDateTime` 형태로 내려준 만료 시간을 브라우저가 시간대 정보 없이 해석하면서, 운영 서버 시간과 사용자 브라우저 시간 사이에 차이가 생길 수 있었기 때문입니다.

이를 해결하기 위해 응답 DTO에 `remainingSeconds`를 추가하고, 프론트엔드는 절대 시간 대신 남은 초를 기준으로 타이머를 표시하도록 변경했습니다.

### 8.5 이메일 인증 완료 후 재발송 방지

이미 인증 완료된 같은 이메일에 대해 인증코드 보내기를 다시 누르면 새 코드가 발급될 수 있었습니다.

같은 이메일과 같은 인증 목적에서 이미 인증이 완료된 경우에는 새 코드를 발급하지 않고, 인증 완료 메시지를 반환하도록 개선했습니다. 사용자가 다른 이메일을 입력하면 기존처럼 새 인증코드 발송이 가능합니다.

### 8.6 Spring Security generated password 경고 제거

가힣는 자체 회원 로그인 + 세션 + 인터셉터 방식으로 인증을 처리하지만, Spring Boot가 기본 개발용 사용자를 자동 생성하면서 운영 로그에 generated password 경고가 출력되었습니다.

기본 `UserDetailsServiceAutoConfiguration`만 제외해 경고를 제거하고, CSRF와 보안 헤더 등 기존 SecurityFilterChain은 유지했습니다.

### 8.7 MySQL 백업 권한 문제

운영 DB 백업 중 `mysqldump`가 tablespace 관련 권한 문제로 실패했습니다.

`--no-tablespaces` 옵션을 추가해 애플리케이션 복구에 필요한 DB 덤프를 정상 생성했습니다.

### 8.8 Nginx 스캐너 경로 차단

공개 서버 노출 후 `/wp-admin`, `/.env`, `/.git`, `/phpmyadmin` 같은 흔한 공격성 경로 요청이 확인되었습니다.

이 요청들이 Spring Boot 앱까지 전달되지 않도록 Nginx에서 404로 차단했습니다.

## 9. 프로젝트를 통해 학습한 점

이 프로젝트를 통해 단순 기능 구현보다 운영 정책을 코드로 표현하는 과정이 중요하다는 점을 학습했습니다.

특히 다음 부분을 중점적으로 다루었습니다.

- Spring MVC 기반 웹 애플리케이션 구조
- 세션 기반 인증과 인터셉터
- Spring Security와 CSRF
- JPA 엔티티 설계
- QueryDSL 기반 검색과 정렬
- Thymeleaf 기반 서버 사이드 렌더링
- 파일 업로드와 다운로드
- 전역 예외 처리와 오류 페이지
- 관리자 로그 설계
- 운영 정책 설계
- dev / prod 환경 분리
- 운영 DB 스키마 검증
- Nginx / systemd / MySQL 기반 실제 배포
- 운영 로그 확인과 문제 원인 추적
- 배포 후 수동 스모크 테스트

## 10. 화면 기반 포트폴리오 구성 초안

별도의 발표 자료나 포트폴리오 문서를 작성할 때는 다음 화면을 캡처하고 기능을 설명할 예정입니다.

1. 홈 / 국가 선택 화면
   - 국가별 커뮤니티 진입 구조 설명

2. 국가 커뮤니티 홈
   - 국가별 게시판과 커뮤니티 컨텍스트 설명

3. 게시글 목록 / 검색 / 정렬
   - QueryDSL 기반 검색, 정렬, 페이지네이션 설명

4. 게시글 상세
   - 댓글, 반응, 첨부파일, 이전/다음 글, 상세 하단 목록 설명

5. 글쓰기 / 첨부파일
   - 이미지 미리보기, 파일 제한, 첨부 다운로드 정책 설명

6. 회원가입 / 이메일 인증
   - 이메일 인증, 타이머, 재발송 제한, 인증 완료 처리 설명

7. 마이페이지
   - 내 글, 프로필 수정, 이메일 변경 인증, 비밀번호 변경, 탈퇴 흐름 설명

8. 관리자 화면
   - 회원/게시글/댓글/신고/로그 관리 설명

9. 신고 및 관리자 조치
   - 신고 접수, 중복 신고 제한, 블라인드/복구, 관리자 로그 설명

10. 배포 구조 다이어그램
   - AWS Lightsail, Nginx, Spring Boot, MySQL, HTTPS, systemd 구성 설명
